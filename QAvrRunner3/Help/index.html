Introduction

QAvrSimulator provides a simple simulation of ATMEGA/AVR architecture systems. In particular it simulates the University of Canterbury FunKit4 development board.

It is designed with simplicity in mind and makes it easy to create a simulation of a simple Embedded circuit with buttons, leds, and ledmat displays. The simulator also allows for serial communication between two copys of QAvrSimulator running on the same computer or with a hardware serial device connected to a serial port on the computer.

Getting Started

QAvrSimulator works by running intel hex files that have been compiled to target an AVR chip. To run a program we need to open a suitable .hex file and select the correct settings for QAvrSimulator to use to run the file.

Steps to run a program

  1. Select hex file.
  2. Select AVR processor type to simulate.
  3. Start the program.

Selecting a hex file

QAvrSimulator requires an intex hex file to run, this will normally be created when you compile your program for deployment and there should be one int the build directory of the compiled program. If you cant find a .hex file in your build directory you will need to use "avr-objdump" to create one.

"avr-objdump" is used to copy the contents of a compiled AVR program into a form that is easy to send to a chip by a programmer. In the case of QAvrSimulator .hex files are a convenient format for the simulator to work with.

To use "avr-objdump", open a terminal window and navigate to where your project has built. Make sure your project has compiled correctly and there are no errors. Find the .out or .elf file, this is the compiled program. Use the following command to create the .hex file.

$avr-objdump -O ihex YourProgram.out YourProgram.hex

If the conversion is successful there will now be a .hex file ready for use by the simulator. Using QAvrSimulator go to "File->Load Hex" and open your hex file.

Select AVR processor type to simulate

Using the drop down list select the processor that your program is targeted to (Note: The University Canterbury Funkit uses the Atmega32u2). It is important that you select the correct chip as there are differences that can make the difference between your program running and not running.

Start the Program

Running the program is controlled using the Play, Pause, and Step buttons on the menu bar. Programs always start in a paused state to allow for stepping through individual instructions, otherwise the program can be unpaused and the program will run normally.

Everytime the Play button is pressed to start QAvrSimulator will load a fresh copy of the program and reset the simulation state. This is usefull if you are making changes to your program while running it in the simulator, every time Play is pressed QAvrSimulator will reload the latest version of your program.

The program will always start/restart in a paused state. This is primarially to help with debugging and it is possible to step through the individual instructions one by one. Using the debug window you will be able to see the processor state change as each machine instruction is executed. In single step mode the machine instructions are disassembled and the resulting assembly instructions are displayed in the "Trace" pane of the debugger.A program can be paused and restarted at any point after it has loaded.

Serial Communications

QAvrSimulator provides options for a serial interface between 2 copies of QAvrSimulator and between QAvrSimulator and real hardware attached to the host computer.

To use the serial interface you must select the port you intend to use and the baud rate required (QAvrSimulator overrides the baud rate defined in your program). These settings will be applied when your program is started or at the next restart if you are already running the program.

QAvrSimulator to QAvrSimulator Communications

To create a communications link between two copies of QAvrSimulator two virtual serial ports are required. These virtual ports must be connected in a Null Modem configuration (Port1 Rx and Tx connects to Port2's Tx and Rx respectively).

Virtual Serial Ports in Linux

This is easy to accomplish in a Linux/Unix based system using a program called "Socat". Socat redirects serial streams of information and can be used to simulate two serial ports. To create two virtual serial ports use the following command

$socat -d -d pty,raw,echo=0 pty,raw,echo=0

Socat will then start and display in the terminal the names of the virtual serial ports which should be of the form "/dev/pts/x" where x is the number of the port. When you set the serial settings use these ports, one for each copy of QAvrSimulator running.

Virtual Serial Ports in Windows

To create a pair of virtual serial ports in Windows you will need to use an additonal program. I suggest "Com0Com". (Com0Com website.) As with socat this will create a Null Modem pair of ports that two copies of QAvrSimulator can use to communicate with eachother.

Additional Serial Settings

It is also possible to make serial transmissions echo back to the sender. This is useful for simulating hardware situations where this is the case (UCFunKit in IR mode for example). There are two options Hardware Echo and Software Echo. Software echo causes to the program running in the simulator to think that it is recieving an echo of any byte it recieves. Hardware echo causes QAvrSimulator to echo recieved bytes back to the device/program it is linked to.

Using Software Echo

With the UCFunKit4 the device UART has a bug which results in the UART imeddiately receiving any byte that it is has sent. It is useful to be able to simulate this so that QAvrSimulator behaves as the UCFunKit would when transmitting information.

To activate Software Echo check the Software Echo option in the Serial Menu. When you start/restart your program the Sofrware Echo setting will be applied.

Using Hardware Echo

When a UCFunKit is hardwired to a serial connection as in the case of connection to a computer (The QAvrSimulator Host) the behaviour of the UCFunKit Uart is modified and the echoing behaviour no longer occurs. To simulate this go to the Serial Menu and check Hardware Echo. With Hardware Echo activated QAvrSimulator will echo any recieved bytes back to the external board, this is sufficient to emulate the normal serial echo behaviours of the UCFunKit.
